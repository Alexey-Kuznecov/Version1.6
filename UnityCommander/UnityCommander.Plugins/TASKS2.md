# 📦 Плагинная библиотека: логирование и структура

## 🧠 Обзор архитектуры логирования

Для обеспечения гибкости и прозрачности, библиотека плагинов использует централизованную систему логирования через `ILoggerService`. Она основана на `NLog`, что позволяет логировать в файлы, консоль или даже в удаленные системы. 

### 🔧 Основные компоненты логирования:

- **ILoggerService** – интерфейс логирования. Определяет базовые методы `LogInfo`, `LogWarning`, `LogError`, `LogDebug`, `LogTrace`.
- **NLogLoggerService** – реализация `ILoggerService` на основе `NLog`. Инициализируется с помощью `NLog.Config.LoggingConfiguration`, определяющей поведение логов (файл, консоль, фильтрация).

### 📄 Класс за классом:

#### ✅ ILoggerService
Контракт для логирования, позволяет инъектировать любой логгер без привязки к конкретной технологии.

#### ✅ NLogLoggerService
Реализация интерфейса `ILoggerService`. Использует `NLog` для записи логов. Создает `Logger` с названием компонента (по типу).

#### ❗Недопустимые ошибки:
- ❌ Не использовать интерфейс `ILoggerService` напрямую в коде — всегда внедрять через зависимости.
- ❌ Жестко не привязывать к `NLog`, чтобы можно было легко заменить логгер в будущем.
- ❌ Не логировать чувствительную информацию (ключи, пароли, приватные пути).

---

## 🚀 Как использовать логирование в проекте:

1. Добавить зависимость `ILoggerService` в конструктор нужного класса.
2. Внедрить `NLogLoggerService` при инициализации, передав нужный `Type`.

```csharp
public class PluginContainer
{
    private readonly ILoggerService _logger;

    public PluginContainer(ILoggerService logger)
    {
        _logger = logger;
    }

    public void LoadPlugins() {
        _logger.LogInfo("Начата загрузка плагинов");
        // логика
    }
}
```

3. Инициализация логгера:
```csharp
var logger = new NLogLoggerService(typeof(PluginContainer));
var container = new PluginContainer(logger);
```

## ✅ Общие задачи:

### 🔷 Архитектура и проектирование:
- Разделить ответственность между `PluginManager` (управление), `PluginContainer` (хранилище), `PluginLoader` (загрузка).
- Все зависимости внедрять через интерфейсы.
- Не допускать циклических зависимостей.
- Изолировать инфраструктурные вещи (логгеры, загрузчики DLL) от логики плагинов.

### 🔷 Ключевые фичи:
- [ ] Возможность загрузки/выгрузки плагинов во время работы.
- [ ] Поддержка метаданных и версионирования плагинов.
- [ ] Автоматическая регистрация зависимостей.
- [ ] Интеграция с системой ролей/доступа.

---

## ✅ Best practices:
- Использовать Dependency Injection.
- Не хранить состояния плагинов вне `PluginContainer`.
- Все доступы к плагинам через `PluginContainer` или `PluginManager`.
- Добавлять подробные логи с указанием имени плагина и действия.
- Делать код тестируемым: не использовать статические зависимости, передавать их через конструкторы.
